# UX patterns

## Назначение файла

Этот файл описывает основные UX-паттерны продуктовой системы.

Он используется вместе с:
- `01_product_foundation/product_philosophy.md`;
- `03_ux/ux_principles.md`;
- `04_ui/ui_base.md`;
- описаниями сервисов из `02_services/`.

Если `ux_principles.md` отвечает на вопрос «как мы думаем», то этот файл отвечает на вопрос «как мы строим интерфейс».

---

# 1. List page

## Назначение

Страница списка — основной паттерн для работы с множеством объектов.

Она нужна, чтобы пользователь мог:
- увидеть общую картину;
- понять приоритеты;
- быстро отфильтровать важные состояния;
- открыть объект;
- выполнить действие.

Список не должен быть просто реестром.

---

## Структура list page

Базовая структура:

1. Header.
2. Summary / mini-analytics.
3. Быстрые фильтры.
4. Список карточек.
5. Нижний плавающий виджет действий.

---

## Header

Header задаёт контекст страницы.

Может содержать:
- название раздела;
- краткое пояснение;
- количество объектов;
- состояние выборки.

Header не содержит primary action страницы, если это действие уже находится в нижнем плавающем виджете.

---

## Summary / mini-analytics

Summary показывает верхнеуровневую картину.

Он отвечает:
- что происходит;
- где проблема;
- сколько объектов в важных состояниях;
- какие зоны требуют внимания.

Summary может содержать:
- KPI;
- распределение по состояниям;
- зоны риска;
- динамику;
- AI-вывод;
- карточки с агрегированными значениями.

### Правило

Элементы summary желательно делать кликабельными, если они могут фильтровать список.

Пример:

Пользователь видит карточку «Просрочено с риском дефолта» → кликает → список показывает только соответствующие объекты.

---

## Быстрые фильтры

Быстрые фильтры находятся на странице.

Форматы:
- toggles;
- chips;
- segmented controls.

Они нужны для самых важных состояний конкретной страницы.

Примеры:
- «Просрочено»;
- «Риск дефолта»;
- «Новые»;
- «Требуют внимания»;
- «Нет риска».

### Правило

Быстрые фильтры — не полный набор параметров.  
Они отражают ключевые пользовательские сценарии страницы.

---

## Расширенный фильтр

Расширенный фильтр открывается в drawer.

Он содержит полный набор фильтров для конкретной страницы.

Может включать:
- статус;
- тип объекта;
- период;
- владельца;
- источник;
- диапазоны значений;
- дополнительные параметры.

### Правило

Быстрые фильтры = быстрый сценарий.  
Расширенный фильтр = точная настройка.

Боковая постоянная панель фильтров в базовом паттерне не используется.

---

## Список карточек

Список строится из карточек.

Таблицы и плотные табличные реестры не используются как основной формат.

### Карточка списка должна содержать

1. Идентификацию:
   - название;
   - ключевой идентификатор.

2. Сигнал:
   - статус;
   - уровень;
   - риск;
   - изменение;
   - проблемное состояние.

3. Ключевые метрики:
   - 1–3 показателя, влияющих на решение.

4. Причину внимания:
   - почему объект важен;
   - что изменилось;
   - что пошло не так;
   - почему его стоит открыть.

Примеры причины внимания:
- «перешёл в зону риска»;
- «рост задолженности за период»;
- «увеличилась просрочка»;
- «ухудшилось финансовое состояние»;
- «появился новый связанный риск».

---

## Приоритизация списка

Список должен отвечать на вопрос:

С чего начать?

Способы приоритизации:
- сортировка по важности;
- группировка;
- визуальный акцент;
- теги состояния;
- причина внимания;
- AI-рекомендация.

Запрещено показывать равнозначный список без приоритета.

---

## Сценарии list page

### Открытие объекта

Если пользователь хочет открыть объект, понять его состояние и принять решение:

list → modal

Примеры:
- открыть риск;
- открыть контрагента;
- открыть продукт;
- открыть инцидент;
- открыть меру.

### Быстрое действие или уточнение

Если пользователь хочет выполнить короткий вспомогательный сценарий:

list → drawer

Примеры:
- открыть расширенный фильтр;
- быстро посмотреть связанную информацию;
- уточнить источник;
- посмотреть короткую историю;
- выполнить лёгкое действие без полного просмотра объекта.

### Главное правило

Modal = открыть объект, фокус, решение.  
Drawer = быстрый шаг, уточнение, вспомогательное действие.

Drawer не является обязательным промежуточным шагом перед modal.

---

## Нижний плавающий виджет действий

На странице глобальные и ключевые действия размещаются в нижнем плавающем виджете.

Он может содержать:
- основное действие страницы;
- кнопку вызова ассистента;
- дополнительные действия.

### Зачем

- действие всегда доступно;
- header не перегружается;
- ассистент остаётся рядом со сценарием;
- страница сохраняет чистую структуру.

---

# 2. Object modal

## Назначение

Object modal — основной паттерн просмотра объекта.

Он используется, когда пользователь хочет:
- открыть объект;
- понять ситуацию;
- увидеть ключевой контекст;
- принять решение;
- при необходимости углубиться в детали.

Объект не открывается в drawer как полный просмотр.

---

## Структура object modal

Модалка объекта состоит из трёх зон:

1. Header.
2. Body.
3. Footer.

Внутри Body:

- Content.
- Meta.

Структура:

```txt
Header
Body
  ├─ Content
  └─ Meta
Footer
```

---

## Header

Header быстро идентифицирует объект.

Может содержать:
- название объекта;
- ключевой идентификатор;
- статус;
- короткие признаки.

Header отвечает на вопрос:

Что это за объект?

Header не должен содержать:
- длинные описания;
- тяжёлую аналитику;
- дублирующие действия;
- смысловые выводы, которые должны быть в Content.

---

## Body

Body — основная зона модалки.

Она делится на:
- Content — смысловой слой;
- Meta — служебный слой.

---

## Content

Content отвечает на вопрос:

Что происходит и что с этим делать?

Содержит:
- summary;
- AI-анализ;
- описание ситуации;
- причины;
- последствия;
- рекомендации;
- ключевые показатели;
- доказательства;
- важные связи;
- связанные блоки.

### Summary внутри Content

Summary — первый смысловой блок Content.

Он должен объяснить:
- что произошло;
- почему это важно;
- насколько это критично;
- какое действие возможно.

Модалка объекта не начинается с набора полей.  
Она начинается с понимания сути.

---

## Meta

Meta отвечает на вопрос:

Что это за объект формально?

Содержит:
- статус;
- тип;
- уровень;
- дату;
- источник;
- владельца;
- идентификаторы;
- короткие служебные признаки.

Meta должна быть:
- компактной;
- вторичной по визуальному весу;
- отделённой от смыслового контента.

### Запрещено помещать в Meta

- выводы;
- рекомендации;
- объяснения;
- длинные описания;
- AI-summary;
- причины и последствия.

---

## Footer

Footer содержит действия модалки.

Может содержать:
- primary action;
- secondary actions;
- reject / destructive action;
- дополнительные действия.

Примеры:
- «Принять»;
- «Отклонить»;
- «Не согласен с оценкой»;
- «Сохранить»;
- «Закрыть».

### Правило

В модалке объекта нет нижнего плавающего виджета.  
Кнопки находятся в footer модалки.

Primary action не дублируется в header.

---

## Drawer внутри object modal

Из модалки может открываться drawer, если нужно:
- посмотреть связанную сущность;
- раскрыть технические детали;
- показать источник;
- открыть историю;
- выполнить короткий вспомогательный шаг.

Drawer не заменяет object modal и не используется для полного просмотра основного объекта.

---

# 3. Drawer pattern

## Назначение

Drawer нужен для короткого шага без потери текущего контекста.

Он используется для:
- расширенного фильтра;
- уточнения информации;
- связанной сущности;
- короткого preview;
- истории;
- служебных деталей;
- промежуточного действия.

---

## Когда использовать drawer

Использовать drawer, если:
- пользователь не должен покидать текущий контекст;
- информация вспомогательная;
- действие короткое;
- полный просмотр объекта не нужен.

---

## Когда не использовать drawer

Не использовать drawer, если:
- пользователь открывает основной объект;
- нужно принять важное решение;
- требуется полноценное чтение объекта;
- нужно много смыслового контента.

В этих случаях использовать modal.

---

# 4. Modal pattern

## Назначение

Modal нужен для фокуса и решения.

Используется для:
- просмотра объекта;
- принятия решения;
- подтверждения;
- работы с содержательным сценарием;
- критического действия.

---

## Когда использовать modal

Использовать modal, если:
- пользователь открывает объект;
- нужно сосредоточиться;
- требуется понять ситуацию;
- есть действия в footer;
- сценарий важнее текущего контекста страницы.

---

## Когда не использовать modal

Не использовать modal, если:
- нужно только быстро уточнить данные;
- действие вспомогательное;
- полный фокус не нужен;
- достаточно drawer.

---

# 5. Summary-first pattern

## Назначение

Summary-first — сквозной паттерн системы.

Он означает, что сложный экран сначала показывает смысл, а потом детали.

---

## Где используется

- list page;
- object modal;
- dashboard;
- AI-анализ;
- карточки состояния;
- экраны мониторинга.

---

## Что содержит summary

Summary может содержать:
- короткий вывод;
- главную метрику;
- изменение;
- зону риска;
- рекомендацию;
- AI-комментарий;
- переход к важному сегменту.

---

## Правила

- Summary должен помогать действовать.
- Summary должен объяснять, куда смотреть дальше.
- Summary не должен быть декоративным.
- Summary не должен дублировать очевидные данные.

---

# 6. AI blocks

## Назначение

AI-блоки помогают пользователю понять ситуацию, объяснить оценку и перейти к действию.

AI в интерфейсе — не отдельная сущность «сбоку», а встроенный слой продукта.

---

## Где может быть AI

AI может быть:
- в summary;
- в карточке;
- в модалке объекта;
- в нижнем плавающем виджете страницы;
- в объяснении оценки;
- в рекомендациях;
- в подсказках;
- в сценарии ассистента.

---

## Правила AI-блоков

AI-блок должен:
- быть привязан к текущему контексту;
- объяснять, а не украшать;
- помогать принять решение;
- давать следующий шаг.

Запрещено:
- использовать AI как абстрактный чат без контекста;
- выводить длинный текст без структуры;
- дублировать очевидное;
- подменять AI-блоком нормальную информационную архитектуру.

---

## Ассистент

Кнопка вызова ассистента на странице находится в нижнем плавающем виджете.

Ассистент должен открываться в контексте:
- текущей страницы;
- выбранного сегмента;
- объекта;
- фильтров;
- активного состояния.

Примеры контекстных сценариев:
- «Объясни, почему вырос риск»;
- «Покажи объекты с ухудшением»;
- «Собери список тех, кто требует внимания»;
- «Объясни изменение оценки».

---

# 7. Filters pattern

## Два типа фильтров

В системе используются два уровня фильтрации:

1. Быстрые фильтры на странице.
2. Расширенный фильтр в drawer.

---

## Быстрые фильтры

Используются для самых важных состояний.

Форматы:
- chips;
- toggles;
- segmented controls.

Они должны быть видны сразу и помогать быстро перейти к частым сценариям.

---

## Расширенный фильтр

Открывается в drawer.

Содержит все возможные параметры фильтрации для конкретной страницы.

---

## Правило для LLM

Не создавать боковую постоянную панель фильтров, если она не задана явно.  
По умолчанию использовать быстрые фильтры на странице и расширенный фильтр в drawer.

---

# 8. Empty, loading, error states

## Empty state

Пустое состояние должно:
- объяснять, почему данных нет;
- предлагать действие;
- не выглядеть как ошибка, если это нормальное состояние.

---

## Loading state

Загрузка должна:
- сохранять структуру страницы;
- не ломать layout;
- показывать skeleton или понятное состояние ожидания.

---

## Error state

Ошибка должна:
- объяснять, что случилось;
- предлагать восстановление;
- не скрывать весь контекст, если часть данных доступна.

---

# 9. Anti-patterns

Запрещённые паттерны:

- начинать сложную страницу со списка без summary;
- использовать таблицу как основной список;
- открывать объект в drawer;
- делать drawer обязательным шагом перед modal;
- помещать primary action страницы в header;
- использовать нижний плавающий виджет внутри object modal;
- смешивать meta и content;
- помещать AI-summary в meta;
- показывать равнозначный список без приоритета;
- использовать AI как абстрактный чат без контекста;
- дублировать действия в нескольких местах без причины;
- добавлять боковую постоянную панель фильтров вместо drawer-фильтра;
- создавать промежуточные шаги без необходимости.

---

# 10. Инструкция для LLM

Перед генерацией интерфейса:

1. Прочитать `ux_principles.md`.
2. Определить тип экрана:
   - list page;
   - object modal;
   - dashboard;
   - form;
   - drawer;
   - modal action.
3. Определить сущность и сценарий.
4. Определить главный сигнал и причину внимания.
5. Выбрать паттерн:
   - list page;
   - object modal;
   - drawer;
   - modal;
   - summary-first;
   - AI block;
   - filters.
6. Разместить действия по правилам:
   - page → нижний плавающий виджет;
   - object modal → footer.
7. Проверить:
   - есть ли summary;
   - есть ли приоритет;
   - правильно ли разделены content и meta;
   - не используется ли таблица;
   - не открыт ли объект в drawer;
   - AI привязан к контексту.

Если правило нарушено, прототип нужно исправить до генерации финального UI.
